# L2 行业/企业场景

> "给工程行业出个方案" / "给中建三局做个方案" —— 宏观行业标签就能出东西。

## 场景定义

| 维度 | 说明 |
|------|------|
| **受众** | 行业中间渠道或大公司接口人（如潘老师、中建三局负责人） |
| **目的** | 让他感受到该课程在其领域内的专业价值，愿意转发给下游 |
| **CTA** | 转发给贵司 XX 负责人，探讨落地可能 |
| **品牌** | 潮汐 AI 品牌完整露出 |

**核心逻辑：** 接收者是中间人，他的目的是把材料转给更具体的人。材料要让他觉得"转发这个对我有利"，同时下游能看懂。

## 输入要求

**必须：** 行业名称 或 公司名（二选一）

**可选：**
- 团队规模
- 已知痛点
- 具体关注方向

## 执行流程

1. WebSearch 搜索该行业/公司的基本信息、AI 应用现状、数字化转型动态
2. `sync` 知识库 → 浏览目录树 → 读「课程体系总览」
3. 根据行业特性，从知识库中选择匹配的素材
4. WebSearch 搜索该行业的 AI 应用案例和生产力数据
5. 生成行业定制材料

> L2 不需要问用户天数，用行业推荐的通用课表。

## 输出格式

| 格式 | 适用场景 | 说明 |
|------|----------|------|
| **HTML**（默认） | 手机转发、微信分享 | 竖版，手机可读，8-12 页 |
| **PDF** | 正式转交 | 可打印、邮件附件 |
| PPT | 内部会议展示 | 横版，适合投影 |

> **PDF 输出注意**：不要用 baoyu → Chromium 链（微信风格排版在 A4 上错乱），用 `references/shiyuan-style-template.md` 的「手写干净 HTML → Chromium PDF」链路。

## 必须包含的组件

| 序号 | 组件 | 用途 | 说明 |
|------|------|------|------|
| 1 | 品牌栏 | 品牌露出 | logo + 标题 + 日期 |
| 2 | Hero 区 | 一句话打动 | 结合行业的价值主张，如"让建筑工程团队效率提升 3-10 倍" |
| 3 | 行业概况 | 建立认知 | 该行业现状 + AI 应用趋势 + 数据 |
| 4 | 背景冲击区 | 紧迫感 | AI 最新动态 + 该行业 AI 案例 + 竞品动态 |
| 5 | 行业痛点 | 引起共鸣 | 3-5 个行业典型痛点（从行业共性提炼，不编造） |
| 6 | 赋能内容概览 | 展示能力 | 课程体系 + 行业适配说明 |
| 7 | 课表框架 | 具体安排 | Unit 级别的课表（每个 Unit 列主题，不展开半天） |
| 8 | 效果展示 | 价值证明 | 2-3 个行业场景的"赋能前 vs 赋能后"对比 |
| 9 | CTA 区 | 行动号召 | "转发给贵司 XX 负责人" |

## 设计要点

- **比 L1 更长**：8-12 页，信息更具体
- **行业语言**：用行业术语，展示"我们懂这个行业"
- **仍以手机可读为主**：竖版，转发后打开即可浏览
- **课表到 Unit 级别**：每个 Unit 列主题和核心内容，不展开半天细节
- **例外**：当用户明确要求\"做课表\"、\"做详细课程安排\"、或类比已有详细课表（如\"和城建类似\"）时，应将每个 Unit 展开到半天级别（半天1-4），包含主题、核心内容、工程设计场景列，形成可直接交付的详细课表

## 关键规则

- 痛点从行业共性提炼，不能凭空编造
- 课表基于知识库素材，行业场景做适配
- 如果输入的是具体公司名，用 WebSearch 查该公司信息增强可信度
- 所有措辞遵循 `references/content-framework.md` 中的措辞规范
- 🔴 **强制图表规则**：每个 L2 方案必须包含 ≥3 张 draw.io 图表，类型按场景灵活选择（IO流程/架构/甘特/对比/业务流程等），少一张即不合格
